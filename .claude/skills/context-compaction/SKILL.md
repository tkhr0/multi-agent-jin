# Context Compaction - コンテキストファイルの統合・圧縮

## Overview

PRマージ後処理で `context/{service}/` 配下のファイルに知見が追記され続けると、ファイルが肥大化し、兵のコンテキストウィンドウを圧迫する。本スキルは肥大化を検知し、情報を欠落させずに統合・圧縮・必要に応じて分割を行う。

## When to Use

- PRマージ後処理で context ファイルに知見を追記した直後
- `context/{service}/base.md` が **300行超** になった場合
- `context/{service}/` 配下の領域別ファイルが **200行超** になった場合
- context ファイルに重複・冗長な記述が目立つと判断した場合

## Trigger（イベント駆動）

大将軍がPRマージ後処理の一環として、知見追記後に以下を実行する:

```bash
# 対象サービスの context ファイル群の行数を確認
wc -l context/{service}/base.md
wc -l context/{service}/*.md
```

閾値を超えたファイルがあれば、本スキルを発動する。

## Instructions

### Step 0: 現状把握

```bash
# 全ファイルの行数を取得し、記録する（コンパクション前の数値）
wc -l context/{service}/*.md
```

出力を「コンパクション前」として記録せよ。完了後に比較する。

### Step 1: 全ファイルの通読

対象サービスの `context/{service}/` 配下の全ファイルを読む。以下の観点でメモを取れ:

- **重複**: 同じトピックが複数ファイル・複数箇所に記載されていないか
- **冗長**: 説明が不必要に長い箇所はないか
- **PR依存**: 特定のPR番号に依存した記述が汎用化できないか
- **陳腐化**: 既に不要になった情報（マージ済みの一時的注意事項等）はないか
- **分類誤り**: 本来別のファイルに属すべき記述はないか

### Step 2: 統合・圧縮

以下のルールに従って圧縮する:

**統合ルール:**
- 同じトピックが複数箇所にある場合、最も詳細な記述を残し、他は削除
- 関連する小さな知見は1つのセクションにまとめよ

**汎用化ルール:**
- `PR #XXX で判明` → 汎用的なルール・注意事項として書き直す
- 例: `PR #498 で XSS 脆弱性を修正` → `JSON-LD 出力時は必ず safeJsonLd ユーティリティを使用せよ`
- PR番号は完全に消す必要はないが、ルールの本文がPR番号なしで理解できる形にせよ

**削除ルール:**
- マージ済みで再発しない一時的な注意事項は削除してよい
- ただし「ハマりやすい落とし穴」として再発の可能性があるものは残せ

**絶対禁止:**
- 有用な知見の欠落。迷ったら残せ

### Step 3: 分割の検討

圧縮後もファイルが閾値を超える場合、分割を検討する:

- `base.md` が300行超 → 領域別ファイルに移動できる記述がないか確認
- 領域別ファイルが200行超 → サブ領域に分割できないか確認
- 既存のどのファイルにも収まらない場合 → 新しい領域別ファイルを作成

**新ファイル作成時の規則:**
- ファイル冒頭に「どのような兵が読むべきか」を明記
- `base.md` から新ファイルへの参照を追加

### Step 4: コンパクション後の検証

```bash
# コンパクション後の行数を取得
wc -l context/{service}/*.md
```

**必須チェック:**
- コンパクション前後の行数を比較し、報告に含めよ
- 削除した情報がないか最終確認（セクション見出しの差分で確認）

### Step 5: wiki への同期（条件付き）

サービスリポジトリに `wiki/` ディレクトリ（git submodule）が存在する場合、コンパクション後の context を wiki に同期する。

```bash
# wiki submodule が存在するか確認
if [ -d "{service_path}/wiki" ]; then
  # context/{service}/ 配下の各ファイルを wiki 向けに整形して wiki/ に書き出す
  # → wiki 内で commit → push
  cd {service_path}/wiki
  git add -A
  git commit -m "docs: コンパクション後の技術知見を同期"
  git push origin master
fi
```

**ルール:**
- `wiki/` が存在しなければスキップ（何もしない）
- このステップは大将軍または軍師が実行する（兵の責務ではない）
- エージェント向け表現は人間向けに言い換えよ（「兵は〜」→「開発者は〜」等）
- context の内容そのものは変えない。表現だけ調整
- メインリポジトリの submodule 参照は更新しない（GitHub Actions が週次で行う）

### Step 6: 報告

大将軍から本陣への報告、または軍師から大将軍への報告に以下を含めよ:

```
コンテキストコンパクション完了。
対象: context/{service}/
コンパクション前: 合計 XXX 行（base.md: XXX行, openapi.md: XXX行, ...）
コンパクション後: 合計 YYY 行（base.md: YYY行, openapi.md: YYY行, ...）
圧縮率: ZZ%
主な変更:
- （統合・圧縮した内容の要約）
- （新規分割したファイルがあれば記載）
wiki 同期: 実施 / スキップ（wiki/ なし）
```

## Guidelines

### 情報の欠落は絶対に許さない

- 圧縮は「同じことを短く書く」であり「消す」ではない
- 迷ったら残せ。過剰に残すほうが、欠落するよりはるかにマシ
- コンパクション前後でセクション見出しを比較し、消えたトピックがないか確認せよ

### base.md の役割を守る

- `base.md` は全兵が必ず読むファイル。ここには共通ルールのみ置く
- 特定領域の詳細知見は領域別ファイルに移動せよ
- base.md が肥大化する主因は「本来領域別に置くべき知見が base に混入している」こと

### 領域別ファイルの冒頭ルール

各領域別ファイルの冒頭には以下を記載:

```markdown
> **対象読者**: {どのような兵が読むべきか}
> 例: OpenAPI 定義の変更・API クライアント生成に関わる兵
```

### PR番号の扱い

- ルールの根拠としてPR番号を残すのは可（例: 末尾に `(ref: PR #XXX)` ）
- ただしルール本文はPR番号なしで意味が通る形で書け
- 古いPR番号の参照が大量にある場合、特に重要なもの以外は削除してよい

## Examples

### Input
PRマージ後処理で `context/{service}/base.md` に知見を追記した結果、320行になった。

### Output
コンパクションを実施:
- 重複していたテーマカスタマイズの記述を1箇所に統合
- `PR #498 で修正した XSS 対策` → `JSON-LD には safeJsonLd を使用` に汎用化
- base.md から OpenAPI 固有の記述を openapi.md に移動
- 結果: base.md 320行 → 245行（23%圧縮）

## Context Directory Structure（参考）

```
context/{service}/
  base.md         # 全兵必読。基本情報・コーディング規約・運用ルール（目標: 300行以内）
  openapi.md      # OpenAPI 定義・API クライアント生成の知見（目標: 200行以内）
  design.md       # テーマ・UI・デザインの知見（目標: 200行以内）
  e2e.md          # E2E テストの知見（目標: 200行以内）
  deploy.md       # デプロイ・インフラの知見（目標: 200行以内）
  {domain}.md     # 必要に応じて追加（目標: 200行以内）
```
