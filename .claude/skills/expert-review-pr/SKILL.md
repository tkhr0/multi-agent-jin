# Expert Review PR — 専門家レビュー付き PR 作成スキル

## Overview

実装完了後、PR を作成する前に以下のプロセスを踏む:

1. **専門家召喚**: 変更内容に応じた複数の専門家ペルソナでコードレビュー
2. **指摘対応**: HIGH は全件修正、MEDIUM は判断して対応、LOW は記録
3. **品質チェック**: lint / format / test を通す
4. **PR 作成**: レビュー結果と対応理由を PR 本文に記載

「セルフレビューの仕組み化」により、人間レビュアーの負荷を下げ、PR の品質を底上げする。

## When to Use

- 機能実装が完了し、PR を作成する前
- バグ修正の PR を作成する前
- リファクタリングの PR を作成する前
- **全ての PR 作成時に使用する**（trivial な修正を除く）

### 使用しなくてよい場合

- ドキュメントのみの変更
- 設定ファイルの軽微な変更（.env, CI config のパラメータ調整等）
- 自動生成ファイルの更新

## Instructions

### Step 1: 変更内容の把握

PR に含まれる変更の全体像を把握する。

```bash
# 変更ファイル一覧
git diff --stat main

# 変更内容の確認
git diff main
```

以下を整理する:
- **変更の目的**: 何を達成するための変更か
- **影響範囲**: どのモジュール / レイヤーに触れるか
- **変更の種類**: 新規追加 / 修正 / 削除 / リファクタ

### Step 2: 専門家の選定

変更内容に基づき、レビューに必要な専門家を選定する。**最低 3名、最大 5名**。

#### コア専門家（変更内容に応じて必ず含める）

| 変更内容 | 必須の専門家 |
|---------|-------------|
| API エンドポイント追加・変更 | API 設計者、セキュリティエンジニア |
| DB スキーマ変更 | DB アーキテクト |
| 認証・認可の変更 | セキュリティエンジニア |
| フロントエンド変更 | フロントエンドアーキテクト |
| テスト追加・変更 | テストエンジニア |
| CI/CD 変更 | DevOps エンジニア |
| パフォーマンスに影響する変更 | パフォーマンスエンジニア |

#### 常駐専門家（全 PR で参加）

| ペルソナ | 観点 |
|---------|------|
| **コード品質レビュアー** | 可読性、命名、DRY 原則、SOLID 原則、コードの一貫性 |

#### 専門家プール

| ペルソナ | 観点 | チェックポイント |
|---------|------|-----------------|
| **セキュリティエンジニア** | 脆弱性・データ漏洩 | インジェクション、認証バイパス、権限昇格、情報漏洩、CSRF、組織境界 |
| **DB アーキテクト** | データモデル・クエリ | N+1、インデックス、トランザクション、マイグレーション安全性 |
| **API 設計者** | RESTful 設計・一貫性 | 命名規約、HTTP メソッド、エラーレスポンス、既存 API との一貫性、OpenAPI 定義 |
| **フロントエンドアーキテクト** | UI/UX・状態管理 | レンダリング効率、状態管理、アクセシビリティ、エラーハンドリング |
| **テストエンジニア** | テスト品質・カバレッジ | テストケースの網羅性、境界値、モック戦略、テストの保守性 |
| **パフォーマンスエンジニア** | 速度・リソース効率 | クエリ最適化、メモリリーク、バンドルサイズ、キャッシュ戦略 |
| **DevOps エンジニア** | CI/CD・デプロイ | ビルド影響、環境変数、後方互換性、ロールバック可能性 |
| **コード品質レビュアー** | 可読性・保守性 | 命名、関数の責務分離、マジックナンバー、コメントの適切さ、型安全性 |

### Step 3: 専門家レビューの実施

各専門家ペルソナになりきり、変更された全ファイルを精読してレビューコメントを生成する。

#### レビューの進め方

1. **ファイルごとに読む**: 変更された各ファイルを先頭から末尾まで読む
2. **専門家の目で見る**: そのペルソナの専門分野に照らして問題を探す
3. **既存コードとの一貫性を確認**: プロジェクトの既存パターンから逸脱していないか
4. **指摘を構造化する**: 以下のフォーマットで記録

#### レビューコメントのフォーマット

```markdown
### {専門家名}

| # | 重要度 | ファイル | 行 | 指摘事項 | 対応 |
|---|--------|---------|-----|---------|------|
| S-1 | HIGH | auth.service.ts | 45 | JWT の有効期限が24時間と長すぎる。トークン漏洩時のリスクが高い | 1時間に短縮 + refresh token で延長 |
| S-2 | MEDIUM | user.controller.ts | 78 | パスワード変更後に既存セッションが無効化されない | 全 refresh token を削除する処理を追加 |
| S-3 | LOW | auth.guard.ts | 12 | エラーメッセージに内部情報が含まれている | 汎用メッセージに変更 |
```

#### 重要度の基準

| 重要度 | 基準 | PR への影響 |
|--------|------|------------|
| **HIGH** | セキュリティリスク、データ不整合、本番障害の可能性 | **修正必須。未修正で PR 作成禁止** |
| **MEDIUM** | 改善が望ましいが、即座の問題にはならない | 対応するか、対応しない理由を明記 |
| **LOW** | コード品質の改善提案、将来の拡張性 | 記録のみ。スコープ外として見送り可 |

### Step 4: 指摘事項の対応

1. **HIGH 指摘**: 全件修正する。修正後に該当コードを再確認
2. **MEDIUM 指摘**: 対応するか判断し、対応しない場合は理由を明記
3. **LOW 指摘**: 記録のみ。PR 本文の「未対応の指摘事項」に残す

修正のコミットは既存コミットとは分けず、**該当する既存コミットに修正を含める**（`git commit --amend` または修正コミットを追加）。

### Step 5: 品質チェック

全修正完了後、以下を実行する。

```bash
# フォーマットチェック
pnpm run format:check

# lint
pnpm run lint

# 型チェック（TypeScript の場合）
pnpm run typecheck  # または tsc --noEmit

# テスト
pnpm run test

# ビルド確認（必要な場合）
pnpm run build
```

**全てパスするまで PR を作成しない。** エラーがあれば修正して再実行。

### Step 6: PR の作成

以下のフォーマットで PR を作成する。

```bash
gh pr create --title "<prefix>: <概要>" --body "$(cat <<'EOF'
## 概要

<何をしたか、1〜3文>

## 変更内容

- <主要な変更点を箇条書き>

## 専門家レビュー結果

### 対応済みの指摘事項

| # | 専門家 | 重要度 | 指摘事項 | 対応内容 |
|---|--------|--------|---------|---------|
| S-1 | セキュリティ | HIGH | JWT有効期限が長すぎる | 1時間に短縮 + refresh token 延長に変更 |
| D-1 | DB | MEDIUM | findMany にインデックスが効かない | product_id にインデックス追加 |

### 未対応の指摘事項

| # | 専門家 | 重要度 | 指摘事項 | 見送り理由 |
|---|--------|--------|---------|-----------|
| C-2 | コード品質 | LOW | ヘルパー関数の抽出 | 現時点では1箇所のみの使用。再利用が発生したら抽出する |

## AI 分析

- **リスク**: <この変更で壊れやすい点>
- **境界条件**: <注意すべきエッジケース>
- **セキュリティ**: <セキュリティ観点の注意点>
- **パフォーマンス**: <パフォーマンス観点の注意点>

## AI開発の振り返り

- **AI活用で効果的だった点:**
- **人間の介入が必要だった点:**
- **改善提案:**
EOF
)"
```

## Examples

### 例1: API エンドポイント追加

**選定した専門家**: セキュリティエンジニア、API 設計者、テストエンジニア、コード品質レビュアー

```markdown
### セキュリティエンジニア
| # | 重要度 | ファイル | 指摘事項 | 対応 |
|---|--------|---------|---------|------|
| S-1 | HIGH | admin-member.controller.ts:34 | organizationId をリクエストから受け取っている。JWT の orgId を使うべき | req.user.currentOrganizationId に変更 |
| S-2 | MEDIUM | admin-member.controller.ts:67 | メンバー削除で自分自身を削除できてしまう | 自分自身の削除を禁止するバリデーション追加 |

### API 設計者
| # | 重要度 | ファイル | 指摘事項 | 対応 |
|---|--------|---------|---------|------|
| A-1 | MEDIUM | admin-member.controller.ts:20 | GET /members のレスポンスにパスワードハッシュが含まれる | select で除外 |
| A-2 | LOW | admin-member.controller.ts:45 | PATCH のレスポンスが void。更新後のオブジェクトを返すべき | 更新後のメンバー情報を返すように変更 |

### テストエンジニア
| # | 重要度 | ファイル | 指摘事項 | 対応 |
|---|--------|---------|---------|------|
| T-1 | HIGH | admin-member.controller.spec.ts | 組織境界テスト（Org A のユーザーが Org B のメンバーを操作できないこと）が欠如 | 組織境界テストケース3件追加 |
```

### 例2: フロントエンド変更

**選定した専門家**: フロントエンドアーキテクト、コード品質レビュアー、テストエンジニア

```markdown
### フロントエンドアーキテクト
| # | 重要度 | ファイル | 指摘事項 | 対応 |
|---|--------|---------|---------|------|
| F-1 | MEDIUM | MemberList.tsx:89 | useEffect 内で毎回 fetch が走る。依存配列に organizationId がない | 依存配列に追加 + SWR/useSWR でキャッシュ |
| F-2 | LOW | MemberForm.tsx:45 | フォームの状態管理が useState 複数で冗長 | 現状のフォーム規模では問題なし。複雑化したら react-hook-form 導入を検討 |
```

## Guidelines

- **全ての PR で最低 3名の専門家をアサインする**。trivial な変更でもコード品質レビュアーは必ず含める
- **HIGH 指摘は全件修正してから PR を作成する**。未修正の HIGH がある状態で PR を作成してはならない
- **MEDIUM の見送りには必ず理由を書く**。「スコープ外」だけでなく、なぜスコープ外と判断したかを記載
- **レビューは変更ファイルだけでなく、影響を受ける周辺コードも確認する**。import 元、呼び出し元、テストファイル
- **プロジェクトの既存パターンとの一貫性を重視する**。新しいパターンを導入する場合はその理由を明記
- **lint / format / test を全て通してから PR を作成する**。CI で落ちると人間レビュアーの時間を無駄にする
- **PR 本文のレビュー結果テーブルは、人間レビュアーへの情報提供**。既にセルフレビュー済みであることを示し、レビューの焦点を明確にする
- **専門家の数は変更の規模に応じて調整する**。小さな修正に5人は過剰、大きな機能追加に3人は不足
- **レビューは網羅的に行う**。「多分大丈夫」で済ませず、各専門家の観点で全ファイルを確認する
