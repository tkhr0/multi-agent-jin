# ADR Writer — Architecture Decision Record 起草スキル

## Overview

設計判断が発生した際に、ADR を書くべきかどうかを判定し、書くべきであれば起草する。「いつ書くか」「何を書くか」「どう書くか」を標準化することで、設計判断の記録漏れを防ぐ。

## When to Use

以下のいずれかに該当する場面で使用する:

- 新しいライブラリ・ツール・フレームワークを選定したとき
- アーキテクチャの構造（レイヤー、モジュール分割、API 設計）を決めたとき
- データモデルを新規作成または大幅変更するとき
- 認証・認可・セキュリティに関する設計判断をしたとき
- 本番障害やレビュー指摘から再発防止策を決めたとき
- フェーズの順序変更やスコープ変更があったとき
- 一時的なコード（デモ、PoC）を追加し、将来削除予定のとき
- 「なぜこうしたのか？」と将来のメンバーが疑問に思いそうな判断をしたとき

| 4 | データベーススキーマの変更（テーブル追加、カラム変更、リレーション変更）を伴うか？ | データモデル |

## トリガーカテゴリ別の詳細

### 1. アーキテクチャ設計（フェーズ開始時）

**タイミング**: フェーズの実装開始前
**トリガー**: 主要サブシステムの構築、新しいレイヤーやモジュール構成の決定

```
例:
- API 基盤設計（エラーハンドリング、ページネーション、OpenAPI）
- マルチテナント基盤（Organization → Product 階層）
- 認証・認可アーキテクチャ（JWT + API Key + RBAC）
```

### 2. ライブラリ・ツール選定

**タイミング**: 導入時（プロアクティブ）または互換性問題発見時（リアクティブ）
**トリガー**: 複数の候補から選択、バージョン制約による非自明な選択

```
例:
- Turborepo vs Nx（モノレポツール）
- Tailwind v3 を選択（v4 は Next.js との互換性問題）
- bcrypt vs Argon2（パスワードハッシュ）
```

### 3. セキュリティ・認証設計

**タイミング**: 認証・認可システムの設計時（必ずプロアクティブに書く）
**トリガー**: 「誰が何をできるか」を定義する全ての判断

```
例:
- JWT の保存場所（Cookie vs localStorage）
- Provider / Tenant 分離と権限モデル
- API Key のハッシュ方式
- 組織 ID の取得ルール（JWT から取得、リクエストから受け取らない）
```

### 4. データモデル変更

**タイミング**: 新しい要件で既存スキーマが対応できないとき
**トリガー**: テーブル追加、リレーション変更、Enum 追加

```
例:
- User-Organization を 1対多 → 多対多に変更
- 記事バージョニングテーブルの追加
- KnowledgeBase モデルの廃止 → Product に統合
```

### 5. インシデント起因の方針変更

**タイミング**: 障害やバグの再発防止策を決めたとき（リアクティブ）
**トリガー**: 構造的な問題（一回限りの typo 修正ではない）

```
例:
- API ルート不一致バグ → Playwright E2E テスト導入を検討（一旦延期 → 再発で導入決定）
- orval 生成コードの不整合 → E2E テスト必須化
```

**重要**: インシデント ADR は「延期」の判断も記録すべき。後で条件を満たしたときに再評価できる。

### 6. プロジェクト順序・スコープ変更

**タイミング**: ステークホルダーの要請や優先度変更時
**トリガー**: フェーズの入れ替え、機能の延期・前倒し

```
例:
- Phase 5（チャットボット）と Phase 7（管理運用機能）の順序入れ替え
- Phase 5.5 の挿入（ユーザー・組織権限設計）
```

### 7. 一時コード・デモ用設計

**タイミング**: 明確な削除予定があるコードを追加するとき
**トリガー**: PoC、デモ、代替サービス待ちの仮実装

```
例:
- チャットボットデモ（askrun 完成後に削除）
  → 削除手順を ADR に明記（対象ファイル、環境変数、依存パッケージ）
```

## Instructions

### Step 1: トリガー判定

チェックリストを走査し、ADR が必要か判定する。

### Step 2: 既存 ADR の確認

```bash
ls docs/adr/
```

- 次の ADR 番号を決定（最大番号 + 1）
- 関連する既存 ADR を特定（置換・廃止する場合は明記）

### Step 3: ADR を起草

以下のテンプレートで起草する:

```markdown
# NNNN: {タイトル}

- **日付:** YYYY-MM-DD
- **ステータス:** 承認
- **関連タスク:** Phase N / PR #XX
- **廃止:** ADR XXXX の {セクション名}（該当する場合のみ）

## コンテキスト

{何を判断する必要があったか。背景・課題・制約条件を記述}

## 判断

{何を決めたか。各判断項目にサブセクションを設ける}

### 1. {判断項目}
{決定内容}

**理由:**
- {なぜこの選択をしたか}

## 却下した代替案

### A. {代替案名}
{何がダメだったか。具体的な理由}

## 影響

- {この判断によって何が変わるか}
- {既存コード・テスト・運用への影響}
- {将来の拡張可能性への影響}
```

### Step 4: レビュー・補足

- **却下した代替案は必ず書く**。「なぜそっちにしなかったか」が ADR の核心
- **影響セクションで既存 ADR との関係を明記**。「ADR 0009 の RBAC セクションを本 ADR で置換」等
- **将来の再評価条件があれば記載**。「Next.js 16 で互換性が解消されたら v4 に移行検討」等

## ADR を書かなくてよい場合

以下は ADR **不要**:

- 単純なバグ修正（typo、off-by-one エラー等）
- ライブラリのパッチバージョンアップ（破壊的変更なし）
- UI の見た目の調整（色、余白、フォントサイズ等）
- テストの追加（テスト戦略自体の変更でなければ）
- リファクタリング（外部から見た振る舞いが変わらなければ）
- ドキュメントの修正

**判断基準**: 「この変更を知らない人がコードを読んだとき、なぜこうなっているか自明か？」 → 自明なら ADR 不要。

## Examples

### 良い ADR タイトルの例

```
0009: 認証・認可アーキテクチャ（JWT + API Key + RBAC）
0011: Admin JWT トークンの Cookie 保存
0013: メディア管理の S3/MinIO 採用
0020: Phase 実行順序の変更（5 ↔ 7）
0023: ユーザー・組織・権限モデルの再設計
0024: Phase 5 チャットボット デモ用設計
```

### 悪い ADR タイトルの例

```
❌ "認証を実装した"         → 「何を判断したか」が不明
❌ "Prisma を使う"          → 選択の文脈がない
❌ "バグを修正した"         → ADR の対象ではない
❌ "Phase 2 の実装"         → 判断ではなく作業の記録
```

## Guidelines

- **プロアクティブに書く**: フェーズ開始時・サブシステム設計時に書くのが理想。実装後に書くと「なぜ」を忘れる
- **却下した代替案が最も重要**: 「なぜ X にしなかったか」が将来の再検討時に最大の価値を持つ
- **短く書く**: 1 ADR = 1 判断テーマ。複数の判断を1つの ADR に詰め込まない（ただし Phase 設計 ADR は例外的に複数判断を含む）
- **ステータスを管理する**: 承認 → 廃止 のライフサイクル。古い ADR は「廃止」にして新 ADR へのリンクを貼る
- **コードと一緒にコミットする**: ADR は実装 PR と同じブランチでコミットし、PR 本文に ADR へのリンクを記載
