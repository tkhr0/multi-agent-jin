# Phase Implementation Plan — フェーズ実装プラン策定スキル

## Overview

新しいフェーズ（大規模機能追加やアーキテクチャ変更）の実装プランを、以下の流れで策定する:

1. 初回プラン作成（詳細設計書ドラフト）
2. 専門家レビュー（複数ペルソナによる多角的レビュー）
3. レビュー反映・吟味（殿との対話で要件確定）
4. ADR 作成（設計判断の記録）
5. 本体 implementation-plan.md への反映

## When to Use

- 「新しいフェーズを追加したい」「大きな機能を設計して」と指示されたとき
- 既存のアーキテクチャに影響する変更を計画するとき
- データモデル・API・認証/認可など横断的な変更を設計するとき
- 「実装プランを作って」「設計書を書いて」と言われたとき

## Instructions

### Phase 1: コンテキスト収集

設計に入る前に、以下を収集・理解する。

1. **既存の実装プラン**: `docs/implementation-plan.md` を通読
2. **関連する既存 ADR**: `docs/adr/` から関連する設計判断を確認
3. **現在のコードベース**: 変更対象のモデル・API・コントローラの現状を探索
4. **殿の要件**: 何を達成したいか、制約条件は何か

```
収集すべき情報:
- 現在のデータモデル（schema.prisma 等）
- 現在の API エンドポイント一覧
- 現在の認証・認可の仕組み
- 関連する既存フェーズとの依存関係
```

### Phase 2: 初回プラン作成（詳細設計書ドラフト）

以下の構成で詳細設計書のドラフトを作成する。

**ファイル**: `docs/phase{N}-detail.md`（または `docs/phase{N}.{M}-detail.md`）

```markdown
# Phase {N}: {タイトル}

## Context
- 背景・課題・目的

## 1. データモデル変更
- 新規 Enum / テーブル
- 既存テーブルの変更
- ER 図
- 権限マトリクス（該当する場合）
- マイグレーション方針

## 2. API 変更
- 新規エンドポイント
- 既存エンドポイントの変更
- Guard / Decorator の追加・変更
- 認証・認可の変更

## 3. フロントエンド変更
- 新規ページ / コンポーネント
- 既存画面の変更

## 4. タスク分割（サブタスク × Wave）
- Wave ごとの依存関係と並列実行計画
- 各サブタスクの diff 目安・ブランチ名

## 5. 対象ファイル（主要）
- 変更ファイル一覧
- 新規ファイル一覧

## 6. 検証方法
- 単体テスト
- 境界テスト（組織境界、権限境界など）
- E2E テスト

## 7. 専門家レビュー記録
（Phase 3 で記入）
```

### Phase 3: 専門家レビュー（多角的レビュー）

**初回プラン作成時に必ず実施する。** 以下の専門家ペルソナを呼び出し、それぞれの観点でプランをレビューさせる。

#### 呼び出す専門家（最低3名）

| ペルソナ | 観点 | チェックポイント |
|---------|------|-----------------|
| **セキュリティエンジニア** | 認証・認可・データ漏洩 | 権限昇格の可能性、組織境界の漏れ、JWT の安全性、インジェクション対策 |
| **DB アーキテクト** | データモデル・パフォーマンス | 正規化の妥当性、インデックス設計、N+1 クエリのリスク、マイグレーション戦略 |
| **API 設計者** | RESTful 設計・一貫性 | エンドポイント命名、HTTPメソッドの適切性、エラーレスポンス、既存 API との一貫性 |
| **フロントエンドアーキテクト** | UI/UX・状態管理 | 画面遷移、状態管理の複雑度、パフォーマンス、アクセシビリティ |
| **テストエンジニア** | テスト戦略・カバレッジ | テストケースの網羅性、境界値テスト、モック戦略 |

ドメインに応じて追加:
- **課金/SaaS 専門家**: マルチテナント設計、プラン制限、メーター計測
- **インフラエンジニア**: デプロイ影響、環境変数、マイグレーション手順
- **パフォーマンスエンジニア**: クエリ最適化、キャッシュ戦略、負荷見積り

#### レビュー実施方法

各専門家ペルソナになりきり、以下の形式でレビューコメントを生成する:

```markdown
## 7. 専門家レビュー記録

### セキュリティエンジニア
| # | 重要度 | 指摘事項 | 対応方針 | 対応状況 |
|---|--------|---------|---------|---------|
| S-1 | HIGH | Admin API で organizationId をリクエストから受け取っている箇所がある。偽装攻撃のリスク | JWT の orgId を使用するルールを明文化 | ✅ 反映済み |
| S-2 | MEDIUM | Provider トークンで Admin API にアクセスできる可能性 | UserTypeGuard で分離 | ✅ 反映済み |

### DB アーキテクト
| # | 重要度 | 指摘事項 | 対応方針 | 対応状況 |
|---|--------|---------|---------|---------|
| D-1 | HIGH | OrganizationUser に複合ユニークがない | @@unique([userId, organizationId]) 追加 | ✅ 反映済み |

### API 設計者
...（同形式）

### 未対応の指摘事項
| # | 重要度 | 指摘事項 | 理由 |
|---|--------|---------|------|
| F-3 | LOW | ダークモード対応 | スコープ外。将来フェーズで対応 |
```

#### 重要度の基準

- **HIGH**: 対応しないとセキュリティリスク、データ不整合、または本番障害の可能性
- **MEDIUM**: 対応が望ましいが、リリースブロッカーではない
- **LOW**: 改善提案。スコープ外として見送り可

### Phase 4: レビュー反映・吟味

1. **HIGH 指摘は全て対応する**。プランに反映し、対応状況を「✅ 反映済み」に更新
2. **MEDIUM 指摘は殿と相談**。対応するかスコープ外とするか判断を仰ぐ
3. **LOW 指摘は記録のみ**。未対応の指摘事項セクションに残す
4. **殿との対話**: 不明確な要件、設計判断の分岐点は殿に質問して確定する

### Phase 5: ADR 作成

設計判断を ADR（Architecture Decision Record）として記録する。

**ファイル**: `docs/adr/NNNN-{slug}.md`

```markdown
# NNNN: {タイトル}

- **日付:** {YYYY-MM-DD}
- **ステータス:** 承認
- **関連タスク:** Phase {N}
- **廃止:** （既存 ADR を置換する場合）ADR XXXX の {セクション名}

## コンテキスト
## 判断
## 却下した代替案
## 影響
```

ADR に含めるべき判断:
- データモデルの設計選択（テーブル分離 vs 単一テーブル、1対多 vs 多対多など）
- 認証・認可のアーキテクチャ変更
- API ルート構成の変更
- 新しい Guard / ミドルウェアパターン

### Phase 6: implementation-plan.md への反映

本体の実装プランを更新する。変更対象セクション:

1. **ER 概要**: 新しいテーブル・リレーションを図に追加
2. **主要テーブル**: テーブル定義の追加・変更
3. **リポジトリ構成**: 新規ディレクトリ・ファイルの追加
4. **アプリケーション構成図**: 新しいアプリ・コントローラの追加
5. **設計ルール / 境界テーブル**: 新しいコントローラ種別の追加
6. **API 設計**: 新規エンドポイントの追加
7. **フェーズ分割**: 新しいフェーズの追加（既存フェーズの間に挿入する場合は N.M 形式）
8. **技術スタック**: 変更がある場合のみ

**注意**: implementation-plan.md は全体の俯瞰文書であり、詳細は `docs/phase{N}-detail.md` に委譲する。implementation-plan.md には概要レベルのみ記載し、`> 詳細設計: docs/phase{N}-detail.md` で参照リンクを貼る。

## Examples

### 入力: 殿の指示

```
「Phase 5 の後にユーザー・組織の権限設計を追加したい。
 提供者と利用者の管理者を区別し、ロール・権限を制御したい」
```

### 出力: 生成物一覧

```
1. docs/phase5.5-detail.md        — 詳細設計書（データモデル、API、テスト計画、専門家レビュー記録）
2. docs/adr/0023-xxx.md           — ADR（設計判断の記録）
3. docs/implementation-plan.md    — 更新（ER図、テーブル、API、フェーズ追加）
```

### 専門家レビューの実例（Phase 5.5 から）

```
セキュリティエンジニア:
  S-1 [HIGH] Admin API で organizationId をリクエストボディから受け取っている
       → JWT の orgId を使用するルールを明文化・全コントローラに適用

DB アーキテクト:
  D-1 [HIGH] User-Organization が1対多だと複数組織所属ができない
       → 多対多に変更（OrganizationUser 中間テーブル）

API 設計者:
  A-1 [MEDIUM] Provider と Admin で認証エンドポイントが重複している
       → 別ルートプレフィックス（/api/provider/v1, /api/admin/v1）で分離

テストエンジニア:
  T-1 [HIGH] 組織境界テストが全エンドポイントに必要
       → 全 Admin/Provider API に組織境界テストケースを追加
```

## Guidelines

- **専門家レビューは省略しない**。初回プラン作成時に必ず実施する。最低3名のペルソナでレビュー
- **HIGH 指摘は全て対応してからプランを確定する**。未対応の HIGH があるプランは承認しない
- **殿への質問は具体的に**。「どうしますか？」ではなく選択肢を提示する（2〜4択）
- **タスク分割は PR 粒度を意識する**。1サブタスク = 1PR、diff 500行目標・1000行超え禁止
- **Wave の依存関係を明確にする**。並列実行できるタスクは並列にし、ボトルネックを最小化
- **implementation-plan.md は俯瞰文書**。詳細を書きすぎない。詳細設計書への参照リンクを貼る
- **ADR は判断の「なぜ」を記録する**。「何をしたか」ではなく「なぜその選択をしたか」「何を却下したか」
- **既存の ADR との関係を明記する**。置換する場合は `廃止:` フィールドに記載
